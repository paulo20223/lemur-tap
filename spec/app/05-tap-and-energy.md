# Энергия

Энергия — единственный расходуемый ресурс действия. Она тратится только на **старт раунда мини-игры «Лови купоны»** (см. [06 — Мини-игра «Купоны»](./06-coupon-game.md)): один старт стоит `coupon_session_cost` энергии (по умолчанию 3600). Тапа в игре нет — энергия больше нигде не списывается.

## Энергия = кулдаун раунда

Так как энергия гейтит только купонный раунд, бар фактически **является таймером кулдауна** этой игры. Числа подобраны так, чтобы раунд был доступен **раз в час**:

- `coupon_session_cost == base_max_energy` (по умолчанию 3600) — один раунд осушает весь бар, накопить «впрок» нельзя.
- `ENERGY_REGEN = 1` энергии/сек ⇒ полная регенерация = `base_max_energy / ENERGY_REGEN = 3600 с = 1 час`.

То есть `время между раундами = coupon_session_cost / energy_regen`. Апгрейды `energy_regen` (+0.5/ур.) укорачивают кулдаун, а `max_energy` (+500/ур.) даёт небольшой банк на несколько раундов — обе ветки сокращают эффективный интервал ниже часа.

## Регенерация энергии (ленивый расчёт)

Энергия не хранится «живой» (без фонового тика). При каждом чтении или мутации сервер пересчитывает текущее значение от `stored` и времени последнего обновления, затем сохраняет результат и новый таймстамп:

```ts
current = Math.min(
  maxEnergy,
  stored + Math.floor((now - energyUpdatedAt) * ENERGY_REGEN),
);
// после расчёта:
stored = current;
energyUpdatedAt = now;
```

`maxEnergy` поднимается веткой апгрейда `maxEnergy`, скорость `ENERGY_REGEN` — веткой `energyRegen`.

## Защита

Старт раунда проверяется против доступной энергии: сервер лениво пересчитывает `current`, требует `current >= coupon_session_cost`, иначе отклоняет (`INSUFFICIENT_ENERGY`). Списание происходит при старте и не возвращается. На эндпоинт навешан rate-limit через Redis.
