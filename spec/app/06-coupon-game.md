# Мини-игра «Лови купоны»

Аркадная 2D-мини-игра и **домашний экран** Lemur Tap (роут `/`). Кольцехвостый лемур с корзиной покупок стоит внизу экрана, сверху падают **купоны брендов** (карточки-тикеты с реальным логотипом). Игрок перетаскивает корзину по горизонтали и ловит купоны — управление полностью на клиенте. Раунд длится **30 секунд**; каждый пойманный купон добавляет свои очки к `score`.

Рендер — лёгкое DOM-поле (`scene2d.ts`), без WebGL/three.js: пул абсолютно позиционированных карточек двигается через `transform`, лемур+корзина — встроенный SVG, едущий по горизонтали за `basketX`. Координаты нормализованы 0..1 по полю, поэтому детерминированное расписание спавна (`engine.ts`) переиспользуется без изменений.

## Геймплей

- **Длительность раунда:** `coupon_session_duration_ms` (по умолчанию 30 000), плюс бонус активной корзины. Эффективная длительность — `effectiveCouponDurationMs(cfg, basketTier) = coupon_session_duration_ms + (baskets[basketTier-1]?.durationBonusMs ?? 0)` (один источник в `economy.ts`; tier 0 = базовая корзина, бонус 0). Сервер возвращает её в `coupon.start` как `durationMs` и использует во всех границах раунда (expiry старта, `earliestFinishMs`, clamp `elapsed`, потолок очков). Корзины покупаются в магазине ([13 — money flows](./13-money-flows.md)): больше времени ловить купоны ⇒ выше `score` и доход, при этом антифрод-потолок очков растёт пропорционально удлинённому раунду.
- **Стоимость:** `coupon_session_cost` энергии на старте (по умолчанию 250). Списывается сервером на `coupon.start`, не возвращается. `coupon.finish` идемпотентен (повтор отдаёт ту же награду).
- **Управление:** позиция корзины нормализована 0..1; перетаскивание pointer/touch двигает её (через `getBoundingClientRect`, clamp 0..1). На поле горизонтальный сдвиг лемура = `(basketX − 0.5) × ширина поля`.
- **Ловля:** купон пойман, когда его падающий y пересекает линию корзины и `|couponX - basketX| <= catchHalfWidth` (полуширина корзины + небольшой запас прощения, ~0.10 в нормализованных). Пойманные купоны проигрывают pop/scale-out и всплывающий `+N`; промахнутые падают вниз и деспавнятся.

## Бренды и редкость

Каждый купон — белая карточка-тикет с **реальным логотипом** бренда (одношаговый SVG-глиф из `logos.ts`, отрисованный акцентным цветом бренда) и бейджем `+N` очков. Глифы инлайнятся как локальные ассеты — путь данных взят из проекта simple-icons (CC0); товарные знаки принадлежат правообладателям и используются номинативно как игровой флейвор. Бренды без доступного глифа (**Pepsi**, **Rolex**) показывают фолбэк — **вордмарк** (название бренда его цветом).

## Жизненный цикл сессии

1. **`coupon.start`** — сервер лениво пересчитывает энергию, проверяет `energy >= coupon_session_cost`, списывает её, создаёт `CouponGameSession` с серверным `seed` и возвращает `{ sessionId, seed, durationMs }`, где `durationMs = effectiveCouponDurationMs(cfg, user.basketTier)` (учитывает активную корзину). Клиент играет ровно `durationMs`.
   - **Незавершённый раунд.** Сессия — единственная активная на пользователя (partial unique index `WHERE status='active'`). Перед списанием сервер разбирает остаток прошлого раунда: сессию с истёкшим окном (`expiresAt < now`) лениво помечает `status='expired'` **без возврата** энергии (шанс finish был израсходован); ещё «живую» сессию (окно не истекло — типично раунд, осиротевший рестартом сервера или обрывом соединения, чей `finish` уже не придёт) помечает `status='abandoned'` и **возвращает** `coupon_session_cost` в бар (с зажимом по `max_energy`), после чего сразу стартует новый раунд. Так зависший раунд не блокирует игру до таймаута. Re-roll бесплатен по сути, но не даёт выгоды: `couponMaxScore` игнорирует `seed`, потолок награды от раскладки не зависит. Поздний `finish` по `abandoned`-сессии → `SESSION_EXPIRED`.
2. **Игра** — клиент строит **детерминированное расписание спавна** из `mulberry32(seed)` (редкость, x, скорость падения, моменты спавна) и играет ~30 секунд, накапливая `score`.
3. **`coupon.finish`** — клиент шлёт `{ sessionId, score }`; сервер проверяет, что сессия активна и не истекла, и что `score ≤ coupon_max_score(...)`.
4. **Награда** — `reward = couponReward(score, coupon_mult, cfg)`, запись в леджер типа `coupon`. Finish не трогает энергию.

## Скоринг

- `score` = Σ очков пойманных купонов (накапливается клиентом, валидируется сервером).
- `couponReward(score, coupon_mult, cfg) = min(coupon_max_coins, floor(score × coupon_coin_per_point × coupon_mult))`.
- `coupon_mult` — эффективный множитель игрока из ветки апгрейда `couponMult` (`effectiveCouponMult(level, cfg) = base_coupon_mult + perLevel × level`, +0.1 за уровень).
- Потолок: `couponMaxScore(seed, elapsedSec, cfg, durationMs = coupon_session_duration_ms) = floor(min(elapsedSec, durationMs / 1000) × coupon_max_points_per_sec)`. Сервер передаёт 4-м аргументом эффективную длительность `effectiveCouponDurationMs(cfg, user.basketTier)`, поэтому корзина поднимает и потолок очков (дефолт параметра сохраняет прежнее поведение для tier 0).

## Буст (расходник)

Одноразовый покупаемый буст, живущий в разделе **Награды → Бусты** (рядом с апгрейдами). Покупается за монеты процедурой `coupon.boost` и разово **доливает энергию на попытку**: энергия дозаправляется на `coupon_boost_energy_grant` (= `coupon_session_cost`, одна попытка), с зажимом по `max_energy`. Поскольку бар держит ровно один раунд, опустошённый бар доливается до полного — игрок может сразу сыграть ещё раз, не дожидаясь регена.

- **Цена:** `coupon_boost_price` монет (по умолчанию 50 ≈ половина типичной выплаты за раунд, а не антифрод-кап `coupon_max_coins` ~3000). Списывается на покупке, запись в леджер типа `coupon_boost`.
- **Дневной лимит:** `coupon_boost_daily_cap` покупок на пользователя за UTC-сутки (по умолчанию 50). Считается по числу строк леджера `coupon_boost` с начала текущих UTC-суток; при достижении — ошибка `coupon_boost_limit` (HTTP 429). Ограничивает объём энергии, докупаемой сверх естественного гейта «один раунд в час».
- Покупка атомарна (проверка лимита + списание монет + долив энергии в одной транзакции); сама игра/награда раунда не меняется.

## Валидация / античит

- `seed` и потолок очков — серверные, клиент не может их подменить. Клиент строит расписание спавна так, чтобы целиться в ≤ ~70 % потолка — идеальный раунд никогда не пробивает границу.
- `score > couponMaxScore(...)` → сессия получает `status = 'rejected'`, награда 0 (`SESSION_REJECTED`).
- Finish раньше минимально допустимого времени → `SESSION_REJECTED`; finish после `expiresAt` → `SESSION_EXPIRED`.
- Повторный `finish` по той же сессии идемпотентен (та же награда, без двойного начисления).

Подробнее — [11 — Защита от читов](./11-anti-cheat.md).

## Баланс

Мини-игра сильнее в early-game и остаётся самостоятельным источником дохода. Типичная выплата скилл-раунда заметно ниже `coupon_max_coins` (~3000) — этот потолок антифрод, а не ожидаемое значение. Формулы и коэффициенты — см. [04 — Экономика](./04-economy.md).
