# Обзор

## Зачем

Сейчас контракт между api и webapp держится на трёх несвязанных слоях: plain-DTO в `@lemur/shared`, `class-validator` в NestJS-контроллерах и ручной fetch-клиент (`apps/webapp/src/api/client.ts`, ~290 строк). Типы и валидация дублируются, рассинхрон ловится только в рантайме.

oRPC сводит это к одному источнику правды: zod-контракт в `shared` → сервер реализует тот же контракт → клиент типизируется из него. Любое расхождение input/output ловится компилятором.

## Цели

- Один контракт на zod в `@lemur/shared` для input/output/ошибок всех процедур.
- Типобезопасный клиент в webapp без ручного маппинга путей и тел.
- Сохранить весь NestJS-рантайм: guards (как middleware), Redis-throttler, Prisma, grammY-бот, сервисы.
- Сохранить текущую семантику: короткоживущий JWT, re-auth на 401, ленивые начисления, целочисленные монеты.

## Нон-голы

- Не переписываем бизнес-логику и чистые функции экономики (`shared/src/economy.ts`).
- Не меняем модель данных (Prisma-схему).
- Не вводим cron/фоновые задачи (инвариант проекта сохраняется).
- Не сохраняем REST-совместимость путей — внешних потребителей REST нет (API приватный, только Telegram miniapp). Поэтому RPC-протокол, не OpenAPI.
- Не трогаем Telegram-бот (отдельный транспорт, вне HTTP-контракта).

## Развилка, которую отвергли

`@orpc/nest` с декоратором `@Implement` регистрирует **HTTP-роут на каждую процедуру** и потребляется через `OpenAPILink` — это OpenAPI-транспорт. Он удобен тем, что переиспользует per-route NestJS-декораторы (`@Public/@Throttle/@CurrentUser`), но несовместим с выбранным RPC-протоколом. Поэтому используем монтирование `RPCHandler` на один Nest-роут, а auth/throttle переносим в oRPC-middleware (см. [03 — Сервер](./03-server.md)).
